Deployment Guide

Local development, Render cloud, Docker, and MT5 on Linux

1. Local Development

1 Environment Setup

# Clone and enter
cd /home/bfc/Desktop/Colpo_Groso_BFC

# Bootstrap (installs apt deps + pip packages + creates venv)
bash setup_env.sh

# Or manually:
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

2 Secrets

cp .env.example .env

Edit .env and set:

VariablePurposeRequired By
DISCORD_WEBHOOK_URLDiscord channel webhook for notificationsv1 Signal Bot

Get a webhook URL from: Discord Server → Settings → Integrations → Webhooks → New Webhook → Copy URL.

3 Verify Installation

source venv/bin/activate
pytest tests/ -v        # should show 375 tests, 373 passed, 2 skipped

4 Run in Simulation Mode

# Single cycle (verify pipeline works)
python trade_bot.py --sim --once

# Continuous simulation
python trade_bot.py --sim

# Dashboard available at http://localhost:8000
# Health check at http://localhost:8000/api/health
Simulation Mode
When --sim is passed (or MT5 is unavailable), MT5Connector uses force_sim=True. It generates synthetic OHLCV data with a bullish trend in the last 80 bars to trigger signals. All logic (Ichimoku, risk, verification, CSV logging) runs identically to production.

2. Render Deployment

The dashboard/ directory contains a standalone FastAPI dashboard deployable to Render's free tier.

1 Render Configuration

File: dashboard/render.yaml

services:
  - type: web
    name: ichibot-dashboard
    runtime: python
    buildCommand: pip install -r requirements.txt
    startCommand: uvicorn app:app --host 0.0.0.0 --port $PORT
    envVars:
      - key: DASHBOARD_SECRET
        sync: false
    plan: free

3 Keep-Alive Configuration

Render free tier spins down after 15 minutes of inactivity. The bot includes a built-in keep-alive that pings the dashboard periodically.

In config/strategy_config.json:

{
  "dashboard": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8000,
    "keep_alive_url": "https://your-app.onrender.com",
    "keep_alive_interval_sec": 600
  }
}

The KeepAlive thread (utils/keep_alive.py) pings {keep_alive_url}/api/health every keep_alive_interval_sec seconds. Set the URL to your Render deployment URL.

4 Remote State Push

When the bot runs on a different server (e.g., a Windows VPS with MT5) than the dashboard (Render), use state push:

python trade_bot.py \
  --dashboard-url https://your-app.onrender.com \
  --dashboard-secret your-secret-token

The StatePusher thread periodically POSTs the bot's full state snapshot to /api/state with Bearer auth.

5 Verify Deployment

python utils/verify_deploy.py --url https://your-app.onrender.com

Runs three checks:

  1. Health check — GET /api/health, expects {"status": "ok"}
  2. Screenshot capture — headless Firefox, saves to logs/dashboard_screenshot.png
  3. Image validation — verifies screenshot has real content (not blank)

3. Docker

A minimal Dockerfile is provided for the standalone dashboard:

File: dashboard/Dockerfile

FROM python:3.10-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY app.py .
EXPOSE 8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
# Build and run
cd dashboard/
docker build -t ichibot-dashboard .
docker run -d -p 8000:8000 \
  -e DASHBOARD_SECRET=your-secret \
  ichibot-dashboard
Note
The Docker image contains only the standalone dashboard (dashboard/app.py). The full trading bot requires MT5 and is not containerized — run it natively on a Windows machine or via Wine on Linux (see below).

4. MT5 on Linux (Wine Bridge)

The mt5linux library communicates with a Windows MT5 terminal running inside Wine via RPyC.

1 One-Time Wine Setup

bash setup_mt5_wine.sh

This installs Wine, .NET Framework, and the MT5 terminal into a Wine prefix.

2 Start the Bridge

# Terminal 1 — keep running
python3 start_mt5_bridge.py
# Listens on localhost:18812

3 Run the Bot

# Terminal 2
source venv/bin/activate
python trade_bot.py    # connects to MT5 via bridge
Import Chain
MetaTrader5 (native, Windows) → mt5linux (Wine/RPyC bridge) → MT5_AVAILABLE=False (simulation fallback). The bot gracefully degrades at each step.

5. Split Deployment Architecture

The recommended production setup separates the bot from the dashboard:

ComponentWhereWhy
Trading Bot (trade_bot.py)Windows VPS with MT5Requires native MT5 terminal
Dashboard (dashboard/app.py)Render (free tier)Web-accessible monitoring

The bot pushes state to the dashboard via StatePusher:

# On the Windows VPS:
python trade_bot.py \
  --dashboard-url https://ichibot-dashboard.onrender.com \
  --dashboard-secret <DASHBOARD_SECRET from Render env>

7. Log Files

All logs are written to the logs/ directory (auto-created on first run):

FileWritten ByContent
logs/bot.logLoguruRotating application log (max 10 MB per file)
logs/trades.csvTradeLoggerAll closed trades with PnL, exit reason, verification
logs/failed_actions.csvFailedActionLoggerFailed trades only — 28 columns for post-mortem analysis
logs/dashboard_screenshot.pngverify_deploy.pyHeadless screenshot for deployment verification
Thread Safety
failed_actions.csv uses filelock for thread-safe writes. BotState uses threading.Lock for all reads/writes since the dashboard accesses state from a different thread.

8. Troubleshooting

IssueCauseFix
Dashboard returns 502 on Render Service spun down (free tier) Set keep_alive_url in config. First request after spin-down takes ~30s.
MT5 connection refused on Linux Bridge not running or Wine MT5 crashed Restart start_mt5_bridge.py. Check Wine prefix with wine64 --version.
No signals in sim mode Session filter blocking (UTC hours) Set session_filter.enabled: false or run during 07-20 UTC.
verify_deploy.py screenshot fails Firefox snap can't write to /tmp Script auto-detects snap and writes to ~/snap/firefox/common/.
Port 8000 already in use Another dashboard instance running Change dashboard.port in strategy_config.json or kill the process.
config validation failed Invalid JSON config values Check ConfigLoader._validate() rules: periods ascending, risk ≤ 5%, RR ≥ 1.0.
